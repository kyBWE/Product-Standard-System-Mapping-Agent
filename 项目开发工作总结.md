## 一、项目目标

将企业产品名称 `product_name` 映射到标准分类体系中的唯一 `category_id`。系统需支持：

1. **RAG 向量匹配**（主方案）：PostgreSQL + pg_trgm + pgvector 粗召回，再精排
2. **PageIndex 树形匹配**（备选）：无向量、按层级推理定位
3. **体系自进化**：同义词反馈环、低置信度扩展建议

### 2. 四种匹配模式总览（Web 可选）

为兼顾准确率、时延与可解释性，系统提供四种可切换模式：

| 模式 | 召回方式 | 精排/决策方式 | 适用场景 | 典型耗时 |
|------|----------|---------------|----------|----------|
| **RAG 向量** | pgvector + pg_trgm 融合 Top-K | DeepSeek 对候选一次打分（必要时） | 默认主链路，准确率优先 | 完全同名约 0.1 s；需精排约 2-10 s |
| **RAG+Rerank** | 同 RAG 向量 | DashScope `qwen3-rerank` | 模糊词较多、希望更快精排 | 需精排约 2-5 s |
| **PageIndex** | 树形索引 + 束搜索 | 规则优先，歧义时才调用 LLM | 索引可命中、强调路径可解释 | 索引命中约 0.1 s；未命中数秒级 |
| **PageIndex 逐层LLM** | 同 PageIndex | 每层均由 LLM 选子节点 | 演示推理过程、对比实验 | 路径越深越慢 |

统一评分口径（RAG 系）：`最终置信度 = 0.4 × 粗召回分 + 0.6 × 精排分`

## 三、主要优化与迭代

本阶段优化围绕三条主线推进：准确率、时延、可解释性。

| 优化主线 | 原问题 | 核心改动 | 当前效果 |
|----------|--------|----------|----------|
| **精排链路优化** | 候选逐条调用 LLM，响应慢 | Top-10 候选改为一次性统一打分；模型切换为 `deepseek-v4-flash` | 精排时延由“数十秒”降至约 2-6 s |
| **召回与检索性能** | 向量检索依赖环境，冷启动慢 | 接入 pgvector + HNSW；提供内存矩阵兜底；服务启动预热 | 向量检索稳定在毫秒级，首条请求卡顿显著降低 |
| **中文 trgm 误报治理** | 中文同义词导致 trgm 假高分 | 取消 trgm 早退；同义词长度打折；类名完全一致优先 | “置信度全 1.0”误判明显减少 |
| **新增 RAG+Rerank 通道** | 仅 LLM 精排，耗时波动较大 | 增加 `qwen3-rerank` 作为平行精排器 | 模糊查询场景下常见更快响应 |
| **PageIndex 重构** | 全树扫描近似暴力匹配 | 改为“索引引导 + 束搜索逐层下降” | 路径可解释、可控，索引命中场景极快 |
| **PageIndex 根节点选择** | 索引未命中时逐根调用 LLM（最坏 56 次） | 改为“一次 LLM 56 选 1”后再单路径搜索 | 极端慢例从分钟级下降到秒级/十几秒级预期 |
| **匹配路径展示优化** | UI 仅显示末尾 3 层，路径信息缺失 | 输出完整路径并标注层级角色 | 演示可读性和复盘能力提升 |

快路径（跳过精排）规则保持：产品名与 Top-1 完全一致；或粗分明显领先且向量分足够高。

### 8. pg_trgm 机制调研与 pg_bigm 方案评估（调研完成，**未实施**）

#### 8.1 当前实现：用的是 PostgreSQL 官方扩展，非手动重写

文本粗召回由 `src/index/trgm_index_manager.py` 实现，**核心相似度来自 PostgreSQL `pg_trgm` 扩展**，不是 Python 手写 trigram 算法：

```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE INDEX ... ON category_texts USING gin (category_name gin_trgm_ops);
SELECT similarity(category_name, $query) AS sim FROM category_texts ...
```

项目在此基础上额外加了业务层 `_effective_trgm_score()`（同义词打折、分类名优先），用于缓解中文误报。

#### 8.2 pg_trgm 如何切分？为何中文几乎不参与？

pg_trgm **不是 jieba 分词**，而是对字符串做 **3-gram（三元字符组）** 滑动窗口，且在本环境（Windows + Conda PG 18）实测：

| 字符串 | `show_trgm` 结果 |
|--------|------------------|
| `测量型实时差分GNSS接收机` | **5 个 trigram，全部来自 `GNSS`** |
| `其他测量型GNSS接收机` | 同上 5 个，**与上完全相同** |
| `测量型GNSS接收机` | 同上 5 个，**与上完全相同** |
| `卫星导航定位天线` | **0 个 trigram（空）** |
| `接收机` / `汽油` | **0 个 trigram（空）** |
| `hello` | 6 个 trigram，正常 |

原因：pg_trgm 默认主要对 **连续 ASCII 字母/数字** 提取 trigram，**CJK 汉字在本环境中不参与 trigram 生成**。因此中英混合产品名的有效信息常退化为几个英文字母片段（如 `GNSS`），汉字差异被忽略。

相似度公式：`similarity(A,B) = |A∩B| / |A∪B|`。当两个类名的 trigram 集合完全一致时，即使汉字完全不同，也会得到 **similarity = 1.0**。

#### 8.3 典型案例：GNSS 接收机系列误报

查询 **`测量型实时差分GNSS接收机`** 时，以下五个节点均出现 **trgm = 1.0**（不合理）：

| 类名 | 类名 raw trgm | 同义词 trgm | 误报原因 |
|------|---------------|-------------|----------|
| 测量型实时差分GNSS接收机 | 1.0 | 1.0 | 完全同名，合理 |
| 其他测量型GNSS接收机 | 1.0 | — | 与查询 **trigram 集合相同**（仅 GNSS 5 元组） |
| 测量型GNSS接收机 | 1.0 | 1.0 | 同上 |
| 卫星导航定位天线 | 0.0 | **1.0**（同义词 `GNSS天线`） | 类名不像，但同义词含 GNSS |
| 卫星导航高精度测地应用平台 | 0.0 | **1.0**（同义词 `GNSS高精度测地应用平台`） | 同上 |

粗召回 Top 4 均为 GNSS 接收机系列且 trgm 全为 1.0，无法靠 trgm 区分细分类，**必须依赖向量检索或 LLM/Rerank 精排**。

链路放大效应：trgm=1.0 → 粗分极高 → 旧逻辑可能跳过精排 → UI 显示多个候选置信度接近 1.0。

#### 8.4 已采取的缓解措施（代码层，保留 pg_trgm）

| 措施 | 说明 |
|------|------|
| 向量融合 | 粗召回始终融合 pgvector，不单靠 trgm 早退 |
| `_effective_trgm_score` | 同义词命中按长度比例 ×0.85 打折；分类名完全相等才给 1.0 |
| 完全同名快路径 | 产品名 = 类名时直接返回，不依赖 trgm 误判 |
| 精排保留 | 多名候选粗分接近时仍走 LLM / Rerank |

#### 8.5 pg_bigm 方案评估（**后续计划**）

曾评估用 PostgreSQL 第三方扩展 **`pg_bigm`** 替代/补充 pg_trgm：

| 对比 | pg_trgm（当前） | pg_bigm（未采用） |
|------|-----------------|-------------------|
| 来源 | PG 自带 contrib | 独立扩展，需单独安装 |
| 切分 | 3-gram，中文本环境≈0 | 2-gram，**中文每字参与** |
| 函数 | `similarity()` | `bigm_similarity()` |
| 索引 | `gin_trgm_ops` | `gin_bigm_ops` |
| Windows 安装 | 零成本 | 类似 pgvector，需预编译包或源码编译 + 重启 PG |
| 代码改动 | — | 中等（换 SQL + 重建索引 + 重调阈值） |

**不实施原因**：

1. Windows + Conda PG 18 环境安装 pg_bigm 有额外成本（与 pgvector 同级）
2. 当前 RAG 主链路已有 **向量 + LLM/Rerank 精排**，GNSS 类 case 可通过精排区分
3. 课程周期内优先保证系统稳定演示，避免再引入新扩展依赖

**结论**：pg_trgm 适合英文拼写模糊匹配，**不适合作为中文语义相似度的唯一依据**；中文产品名场景应以 **向量语义检索为主、trgm 为辅**。pg_bigm 是更合理的文本扩展方向，但本项目**调研后决定暂不迁移**。

## 四、系统架构（当前）

### 1. RAG 主链路（默认）

```text
输入产品名
  -> 粗召回 Top-K（pgvector + pg_trgm 融合）
  -> 快路径判断（同名直返 / 明显领先直返）
  -> 精排（二选一：DeepSeek 或 qwen3-rerank）
  -> 输出 category_id + 置信度 + 候选解释
```

### 2. PageIndex 备选链路（可解释）

```text
输入产品名
  -> 树形推理索引 lookup_index（exact / partial）
  -> 束搜索逐层下降（规则 -> 索引引导 -> LLM 消歧）
  -> 索引未命中时：LLM 一次性多根择一
  -> 输出 matched_category_id + 完整匹配路径
```

### 3. 链路选择策略

- 线上默认推荐 RAG 向量或 RAG+Rerank，优先保证整体准确率与稳定时延。
- PageIndex 适合演示“逐层决策路径”，在索引命中时可达到极低时延。
- 四种模式可在 Web 端切换，用于对比不同场景下的速度与可解释性。

## 九、遇到的问题与解决

| 现象 | 根因 | 处理方案 | 当前状态 |
|------|------|----------|----------|
| 首条请求明显偏慢 | ONNX 与索引冷启动 | 服务启动阶段执行预热 | 已解决 |
| 无 pgvector 时检索变慢 | 逐条向量循环计算 | 使用归一化内存矩阵批量点积 | 已解决 |
| RAG+Rerank 耗时异常偏大 | 服务未重启导致路由走错 | 重启服务并校验 `engine` | 已解决 |
| 候选置信度接近 1.0 且误判 | pg_trgm 中文误报放大 | 向量融合 + 同义词打折 + 同名优先 | 已解决 |
| 体系内同名词仍调用 LLM | 早期快路径判定不够强 | 加入“完全同名直接返回”规则 | 已解决 |
| PageIndex 类似暴力遍历 | 旧逻辑全树扫描 | 改为索引引导 + 束搜索 | 已解决 |
| 匹配路径展示不完整 | 仅截取路径末 3 层 | 返回并渲染完整路径 | 已解决 |
| PageIndex 极端慢例（龟苓膏） | 未命中时按根逐次调用 LLM | 改为“多根一次性择一” | 已完成修复 |
| 中文短词 trgm 为 0 | 本环境 CJK trigram 不参与 | trgm 仅作辅助手段，主依赖向量/精排 | 已规避 |

## 十、后续可扩展方向

1. **建立标准化评测集（优先级 P0）**：构建可复现黄金集，统一评估四种模式在准确率、召回率、平均时延、P95 时延上的表现。
2. **打通单条匹配自进化闭环（优先级 P0）**：在 Web `/api/match` 完成后直接触发 `SelfEvolveScheduler.process_match_result`，缩短反馈到生效的周期。
3. **增强 PageIndex 未命中策略（优先级 P1）**：在 LLM 消歧前增加关键词缩根和候选根重排，减少“纯语义硬猜”。
4. **扩展建议审核流（优先级 P1）**：将低置信结果沉淀为待审核新类目建议，补齐“建议-审核-入库”闭环。
5. **中文文本检索增强（优先级 P2）**：在稳定版本上评估 pg_bigm 或分词+n-gram 方案，作为 pg_trgm 的补强而非替代。
6. **CLI 与 Web 模式完全对齐（优先级 P2）**：补齐 `rag_rerank`、`pageindex_force_llm` 的命令行参数，便于离线批量实验。
