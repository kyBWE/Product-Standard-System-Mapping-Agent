## 一、低准确率根因分析

对错误映射与测试集 GT 对比后结论：

1. **评测口径过严**：仅 `category_id` 完全相等；同义叶、父子、设备树↔产业链双树一律算错。
2. **测试集 GT 质量问题（LLM200）**：体系中已存在「产品名同名」的叶节点，但 GT 标到兄弟类/上位类。另有重复产品名、过细/可疑 GT。
3. **真错配仍存在**：短名/缩写歧义、职能近邻误导等。
4. **快通道高置信误命中（修正 GT 前）**：错误侧平均置信度常高于正确侧；大量 &lt;30ms、0.98 分打到「同名叶」（在旧 GT 下被记为错）。

## 二、已落地的修改

### 2.1 宽松评测指标

- 模块：`src/orchestration/eval_scoring.py`
  - **strict**：ID 全等
  - **near**：同父兄弟，或父子（层差 ≤2）
  - **soft**：near，或同子树 LCA≥3
- LLM200：strict 40.9% → soft **63.1%**

### 2.2 大模型合理性兜底（评测侧）

- `LLMAdapter.judge_mapping_reasonable`：对 soft 未命中且有预测的样本，判断「预测路径对企业产品是否合理」（允许双树/同义/近邻；明显错配判否）。
- 脚本：`rescore_eval_llm_judge.py`
- 合理则记 `correct_llm_soft`；无预测（NO_MATCH）不送 LLM，仍算未命中。
- LLM200：soft 63.1% → llm_soft **81.3%**（72 条待判，采纳 36）；LLMv2：50.9% → **70.4%**（72 条待判，采纳 31）

## 六、后续建议

1. 评测默认以 **strict / soft / llm_soft** 三档报告，避免单一严格 ID 误导。
2. 继续清洗 GT：对「双树同义」「制品/原料双视角」人工或规则对齐。
3. 匹配侧增强：短名/缩写消歧、场景同义词（抑尘/包装等），降低真错配；若要对齐历史 ~50%，需恢复/审计同义词增强是否对测试集泄漏。
4. 全量重建中断时，统一用「跳过已有 ID」续跑脚本，避免误清空。
5. **待做（已方案对齐、尚未跑脚本）**：在 **修正后 GT** + full 索引上，扫描不同 `coarse_top_k`（建议 5/10/20/50）：
   - **Recall@K**：GT 是否进入融合粗召回 Top-K
   - **精排后 Accuracy@K**：同一候选上跑 rerank 的严格准确率
   - 附粗排 Top-1；注意配置 `rerank.top_n: 10` 可能截断精排视野；主指标用 strict，soft 作辅表
   - 现有 `*_full.json` 未存完整候选列表，**不能**反推 Recall@K，需单独重跑粗召回

6. **LLM200 新 GT（refined）**：见下方第九节；评测可用 `test_set_llm_200_refined.json`，**勿改 v2**。
